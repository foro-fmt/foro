# Releasing foro

foro uses [cargo-dist](https://github.com/nahco314/cargo-dist) for releases.
Pushing a version tag triggers the Release CI, which builds binaries for all
target platforms and creates a GitHub Release automatically.

Release notes are not auto-generated anymore. Before pushing the release tag,
an AI agent must write end-user release notes into `RELEASE_NOTES.md`, and the
file must include the AI disclaimer line. The Release workflow fails if
`RELEASE_NOTES.md` is missing, still contains placeholder text, or omits the
required disclaimer. The final GitHub Release body is:
- the AI-authored summary from `RELEASE_NOTES.md`
- followed by cargo-dist's generated install/download section

## Checklist

1. **Check dependencies**
   - If `dll-pack` was switched to a local path dep during development, restore it to a git rev:
     ```toml
     dll-pack = { version = "0.3.0", git = "https://github.com/foro-fmt/dll-pack", rev = "<commit-sha>" }
     ```
   - Verify `cargo build` succeeds.

2. **Run tests**
   ```bash
   cargo test --test cli_install
   cargo test --test cli_format -- --skip test_cli_format_cpp   # cpp tests require a compatible GLIBC
   cargo test --test cli_daemon
   ```

3. **Bump the version**
   Edit `version` in `Cargo.toml` (SemVer):
   - New backward-compatible feature → minor bump (e.g. `0.2.x` → `0.3.0`)
   - Bug fixes only → patch bump (e.g. `0.3.0` → `0.3.1`)

   ```bash
   # After editing Cargo.toml:
   cargo build           # updates Cargo.lock
   git add Cargo.toml Cargo.lock
   git commit -m "chore: bump version to X.Y.Z"
   git push origin main
   ```

4. **Write `RELEASE_NOTES.md`**
   Write release notes for end users of foro. Keep them short and focused on
   user-visible changes since the previous release.

   Include:
   - 2-5 notable user-facing changes or fixes
   - a comparison link for full details
   - the required AI disclaimer line at the end

   Do not include:
   - CI details
   - build system or packaging internals
   - daemon/client refactors unless they changed user-visible behavior

   Template:
   ```markdown
   Highlights in foro X.Y.Z:

   - ...
   - ...

   Full change list: https://github.com/foro-fmt/foro/compare/<previous-tag>...X.Y.Z

   ---
   *This release and summary were automatically generated by an AI agent.*
   ```

   Then commit it before tagging:
   ```bash
   git add RELEASE_NOTES.md
   git commit -m "docs: write release notes for X.Y.Z"
   git push origin main
   ```

5. **Push the tag**
   ```bash
   git tag X.Y.Z
   git push origin X.Y.Z
   ```
   This triggers the Release CI (`.github/workflows/release.yml`).

6. **Monitor CI**
   ```bash
   gh run list --repo foro-fmt/foro --limit 5
   ```
   Once all jobs are green, the GitHub Release is created automatically.

## Notes

- Tag names have no `v` prefix (e.g. `0.3.0`). `v0.3.0` also works but prior
  releases use the bare form, so keep it consistent.
- The release body comes from `RELEASE_NOTES.md`, not from cargo-dist's
  auto-generated git log summary, but cargo-dist's generated install/download
  section is appended after it.
- The required disclaimer line is:
  `*This release and summary were automatically generated by an AI agent.*`
- cargo-dist configuration lives in `dist.toml`.
- If plugin URLs in `default_config.json` need updating after a release, see
  `./MAINTAIN_PLUGINS.md`.
